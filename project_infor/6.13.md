# 1号任务书：句级语义拦截网关开发指南
负责主服务升级、句子切分、语义检测调度、整句/整段拦截与审计

## 一、你的任务定位
你负责把第一阶段的“关键词检测 + 脱敏转发”网关，升级为第二阶段的“句级语义保密拦截”网关。你的工作重点不是判断一句话是否保密，而是把整个流程串起来：用户输入进来后切成句子，调用规则检测、语义检测和事实库匹配，最后根据策略决定是否调用 LLM。
本阶段最重要的变化：如果某一句被判断为保密句，高风险场景下直接拦截整个请求，不再做敏感词替换，不再把这句话发给 LLM。
### 共同约束
- 本阶段所有数据必须为模拟数据。项目名、客户名、人员名、系统名、时间、结论、漏洞、报价、部署计划都用虚构内容。
- 严禁把真实涉密材料、真实客户资料、真实内部会议纪要、真实系统缺陷写入代码、测试集、日志或向量库。
- 语义级保密句识别到高风险后，默认直接拦截整个请求，不做局部词替换，也不把原句发给 LLM。
- 保密事实库是给网关检测用的，不是给 LLM 回答问题用的 RAG 知识库，必须单独建设、单独管理、单独审计。
- 日志里默认不保存高风险原文；需要定位问题时保存 request_id、句子 hash、风险类型、模型分数和动作结果。

## 三、第一步：实现句子切分
为什么要切句：甲方关注的是“某一句话本身是保密事实”。如果整段一起判断，既不好解释，也不好定位风险。切句后，每句话都有自己的风险分、风险类型和动作。
### 切分规则
- 按中文句号、问号、感叹号、分号、英文句号、问号、感叹号、换行进行切分。
- 保留原句文本，不做脱敏，不修改内容。
- 短句也要保留，例如“请帮我润色：”虽然不敏感，但要作为低风险句子参与返回。
- 每个句子生成 sentence_id，例如 s1、s2、s3。
- 对空白句子进行过滤。
### 代码骨架
`# core/sentence_splitter.py`
`import re`
`from typing import List, Dict`

`SPLIT_PATTERN = re.compile(r'([^。！？!?;；`
`]+[。！？!?;；]?)')`

`def split_sentences(text: str) -> List[Dict]:`
`text = (text or '').strip()`
`if not text:`
`return []`

`parts = SPLIT_PATTERN.findall(text)`
`results = []`
`for idx, part in enumerate(parts, start=1):`
`sentence = part.strip()`
`if not sentence:`
`continue`
`results.append({`
`'sentence_id': f's{idx}',`
`'text': sentence`
`})`
`return results`

## 四、第二步：统一调用语义检测
3号会提供 detect_confidential_sentence(sentence: str) 接口。你不需要关心它内部是 LLM 分类器还是向量库匹配，你只要按接口调用并汇总结果。
### 调用接口样例
`# core/guard_orchestrator.py`
`from core.sentence_splitter import split_sentences`
`from semantic.semantic_detector import detect_confidential_sentence`
`from policy.semantic_decision import make_semantic_decision`
`from policy.audit_logger import write_audit_log`
`from llm.client import call_llm`

`def handle_chat(request: dict) -> dict:`
`request_id = request.get('request_id') or generate_request_id()`
`text = request.get('text', '')`
`user_id = request.get('user_id', 'demo_user')`

`sentences = split_sentences(text)`
`sentence_results = []`

`for item in sentences:`
`semantic_result = detect_confidential_sentence(item['text'])`
`sentence_results.append({`
`'sentence_id': item['sentence_id'],`
`'text': item['text'],`
`'semantic_result': semantic_result`
`})`

`decision = make_semantic_decision(sentence_results, mode='strict')`
`write_audit_log(request_id, user_id, sentence_results, decision)`

`if decision['action'] == 'block':`
`return {`
`'request_id': request_id,`
`'action': 'block',`
`'message': '检测到疑似内部保密事实，本次请求已拦截，未发送至模型。',`
`'risk_score': decision['risk_score'],`
`'sentence_results': decision['public_sentence_results']`
`}`

`llm_response = call_llm(text)`
`return {`
`'request_id': request_id,`
`'action': 'allow',`
`'risk_score': decision['risk_score'],`
`'llm_response': llm_response`
`}`
### 注意事项
- 在 block 情况下，必须确认没有调用 call_llm。可以在日志中打印 “LLM_NOT_CALLED”。
- 返回给用户时，不要重复展示高风险原句，避免二次泄露。
- 内部调试阶段可以在本地控制台显示原句，但交付演示时必须关闭原句输出。
- 如果 semantic_detector 报错，保密场景建议 fail-closed：返回“检测服务异常，暂不放行”。

## 六、第四步：审计日志设计
审计日志要能追踪系统决策，但不能成为新的泄密点。高风险原句不要写入日志，保存 hash 即可。
`# policy/audit_logger.py`
`import hashlib`
`import json`
`from datetime import datetime`

`def text_hash(text: str) -> str:`
`return hashlib.sha256(text.encode('utf-8')).hexdigest()`

`def write_audit_log(request_id, user_id, sentence_results, decision):`
`safe_sentences = []`
`for item in sentence_results:`
`text = item['text']`
`sem = item['semantic_result']`
`safe_sentences.append({`
`'sentence_id': item['sentence_id'],`
`'text_hash': text_hash(text),`
`'risk_score': sem.get('risk_score', 0),`
`'risk_types': sem.get('risk_types', []),`
`'is_confidential_sentence': sem.get('is_confidential_sentence', False),`
`'suggested_action': sem.get('suggested_action', 'allow')`
`})`

`record = {`
`'time': datetime.utcnow().isoformat(),`
`'request_id': request_id,`
`'user_id': user_id,`
`'final_action': decision['action'],`
`'risk_score': decision['risk_score'],`
`'sentences': safe_sentences`
`}`

`with open('audit_semantic.log', 'a', encoding='utf-8') as f:`
`f.write(json.dumps(record, ensure_ascii=False) + '`
`')`

## 十、常见问题

## 二、保密事实库是什么
第一阶段的敏感词库是一堆词，例如“手机号、客户名、项目名”。第二阶段的保密事实库是一组句子级事实，例如“某方案已通过内部评审，预计下季度试点”。它关注的是句子表达的事实关系，而不是某一个词。

## 九、第七步：数据校验脚本

`# scripts/validate_fact_schema.py`
`import json`
`from pathlib import Path`

`ALLOWED_TYPES = {`
`'project_progress', 'technical_solution', 'system_architecture',`
`'vulnerability_status', 'customer_intention', 'pricing_strategy',`
`'internal_decision', 'personnel_arrangement', 'procurement_plan',`
`'deployment_plan', 'evaluation_result', 'incident_information'`
`}`

`def validate(path='data/simulated/confidential_facts.jsonl'):`
`seen = set()`
`for line_no, line in enumerate(Path(path).open(encoding='utf-8'), start=1):`
`item = json.loads(line)`
`assert item['fact_id'] not in seen, f'duplicate fact_id at line {line_no}'`
`seen.add(item['fact_id'])`
`assert item['fact_type'] in ALLOWED_TYPES, f'invalid type at line {line_no}'`
`assert item['source'] == 'simulation', f'source must be simulation at line {line_no}'`
`assert item.get('fact_text'), f'missing fact_text at line {line_no}'`
`assert isinstance(item.get('paraphrases'), list), f'paraphrases must be list at line {line_no}'`
`print(f'OK: {len(seen)} facts validated')`

## 三、整体检测思路
`输入一个句子 sentence`
`↓`
`A. 向量相似度匹配`
`- sentence 生成 embedding`
`- 到模拟保密事实库中检索 TopK`
`- 如果相似度超过阈值，判为疑似保密事实`
`↓`
`B. LLM 语义分类器`
`- 判断句子是否表达内部未公开事实`
`- 返回风险类型、置信度、原因`
`↓`
`C. 融合决策`
`- 相似度高：高风险`
`- 分类器高置信：高风险`
`- 两者中等：中高风险`
`- 两者都低：放行`
`↓`
`输出统一 detect_confidential_sentence 结果`

## 五、第二步：实现向量相似度匹配
向量匹配用于解决“没有相同关键词，但语义接近”的问题。第一版可以用任意中文/多语言 embedding 模型。如果本地环境还没准备好，可以先用现有 embedding API；如果能本地部署，推荐使用小型中文 embedding 模型。
### 开发建议
- 先做离线建库：把 simulated facts 和 paraphrases 全部转成向量。
- 用户输入一句话时，也转成向量。
- 用余弦相似度计算最相近的 TopK。
- 第一版不追求极致性能，可以先用 numpy 全量计算；数据超过几万条后再换 FAISS。
- 阈值先设为 0.78，后续由 4号根据测试集调参。
### 简化版代码骨架
`# semantic/embedding_matcher.py`
`import numpy as np`
`from typing import List, Dict`

`def cosine(a, b):`
`a = np.array(a)`
`b = np.array(b)`
`return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8))`

`class EmbeddingMatcher:`
`def __init__(self, embedding_client, threshold=0.78):`
`self.embedding_client = embedding_client`
`self.threshold = threshold`
`self.rows = []`
`self.embeddings = []`

`def build(self, rows: List[Dict]):`
`self.rows = rows`
`texts = [r['text'] for r in rows]`
`self.embeddings = self.embedding_client.embed(texts)`

`def search(self, sentence: str, top_k=3) -> Dict:`
`query_vec = self.embedding_client.embed([sentence])[0]`
`scores = [cosine(query_vec, vec) for vec in self.embeddings]`
`order = np.argsort(scores)[::-1][:top_k]`
`hits = []`
`for idx in order:`
`row = self.rows[int(idx)]`
`hits.append({`
`'fact_id': row['fact_id'],`
`'fact_type': row['fact_type'],`
`'text_type': row['text_type'],`
`'similarity': scores[int(idx)]`
`})`
`best = hits[0] if hits else None`
`return {`
`'matched': bool(best and best['similarity'] >= self.threshold),`
`'top_hit': best,`
`'top_k': hits`
`}`

`# semantic/mock_embedding_client.py`
`import hashlib`
`import numpy as np`

`class MockEmbeddingClient:`
`def embed(self, texts):`
`result = []`
`for text in texts:`
`seed = int(hashlib.md5(text.encode('utf-8')).hexdigest()[:8], 16)`
`rng = np.random.default_rng(seed)`
`vec = rng.normal(size=128)`
`result.append(vec / (np.linalg.norm(vec) + 1e-8))`
`return result`

`# semantic/semantic_detector.py`
`from semantic.fact_store import load_facts, expand_fact_texts`
`from semantic.embedding_matcher import EmbeddingMatcher`
`from semantic.llm_semantic_classifier import LLMSemanticClassifier`

`class SemanticDetector:`
`def __init__(self, embedding_matcher: EmbeddingMatcher, classifier: LLMSemanticClassifier):`
`self.embedding_matcher = embedding_matcher`
`self.classifier = classifier`

`def detect(self, sentence: str) -> dict:`
`sim_result = self.embedding_matcher.search(sentence, top_k=3)`
`cls_result = self.classifier.classify(sentence)`

`sim_score = 0.0`
`sim_type = []`
`evidence = []`

`if sim_result.get('top_hit'):`
`top = sim_result['top_hit']`
`sim_score = top['similarity']`
`sim_type = [top['fact_type']]`
`evidence.append({`
`'type': 'fact_similarity',`
`'matched_fact_id': top['fact_id'],`
`'similarity': sim_score`
`})`

`cls_score = cls_result.get('risk_score', 0.0)`
`evidence.append({`
`'type': 'semantic_classifier',`
`'confidence': cls_score`
`})`

`final_score = max(sim_score, cls_score)`
`risk_types = list(set(sim_type + cls_result.get('risk_types', [])))`
`is_conf = final_score >= 0.75`

`return {`
`'is_confidential_sentence': is_conf,`
`'risk_score': round(final_score, 4),`
`'risk_types': risk_types,`
`'suggested_action': 'block_sentence' if is_conf else 'allow',`
`'evidence': evidence`
`}`

## 七、第四步：自动化评估脚本
评估脚本调用 1号的 /chat 接口或直接调用 3号 detect_confidential_sentence 接口。第一版建议两种都支持：单测算法时直调 3号，全链路验收时调 1号 HTTP 接口。
`# scripts/evaluate_semantic_guard.py`
`import json`
`import requests`
`from pathlib import Path`

`API = 'http://127.0.0.1:8000/chat'`

`def load_cases(path):`
`with Path(path).open(encoding='utf-8') as f:`
`for line in f:`
`if line.strip():`
`yield json.loads(line)`

`def predict_action(text):`
`resp = requests.post(API, json={'user_id': 'eval_user', 'text': text}, timeout=20)`
`data = resp.json()`
`return data.get('action'), data`

`def main():`
`cases = list(load_cases('data/simulated/semantic_test_cases.jsonl'))`
`total = len(cases)`
`correct = 0`
`stats = {}`
`errors = []`

`for case in cases:`
`pred_action, raw = predict_action(case['text'])`
`expected = case['expected_action']`
`ok = pred_action == expected`
`correct += int(ok)`
`cat = case['category']`
`stats.setdefault(cat, {'total': 0, 'correct': 0})`
`stats[cat]['total'] += 1`
`stats[cat]['correct'] += int(ok)`
`if not ok:`
`errors.append({`
`'case_id': case['case_id'],`
`'category': cat,`
`'text': case['text'],`
`'expected': expected,`
`'predicted': pred_action,`
`'raw': raw`
`})`

`print('overall_accuracy', correct / total if total else 0)`
`for cat, s in stats.items():`
`print(cat, s['correct'] / s['total'])`

`Path('reports').mkdir(exist_ok=True)`
`Path('reports/semantic_errors.json').write_text(`
`json.dumps(errors, ensure_ascii=False, indent=2), encoding='utf-8'`
`)`

`if __name__ == '__main__':`
`main()`

## 十四、常见问题
- 问：测试集中能不能用真实项目句子？答：不能，本阶段全部模拟。
- 问：准确率高就行吗？答：不行，要单独看 keywordless_recall 和 false_positive_rate。
- 问：为什么 hard negative 这么重要？答：没有 hard negative，系统会把“请解释内部评审”这种正常问题误拦。
- 问：甲方演示重点是什么？答：重点展示没有项目名、客户名、敏感词的句子也能被拦截。
- 问：发现漏报怎么办？答：先归类，再反馈：数据缺失找 2号，算法阈值找 3号，接口问题找 1号。
